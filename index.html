Encoder-Decoder Model

Encoder-Decoder (Seq2Seq) Model

A neural architecture used when input and output sequences can have different lengths and don't align word-to-word. Also called sequence-to-sequence networks. Key applications: machine translation, summarization, dialogue, question answering.

Why Not Simple RNN?

In simple tasks like part-of-speech tagging, each input token maps to exactly one output tag — local information suffices. But in machine translation, a verb may appear at the beginning in one language and at the end in another. The alignment is highly indirect, requiring the encoder-decoder architecture.

Three Core Components

📥

1. Encoder

Accepts input sequence x1:n, generates contextualized hidden representations h1:n. Can be LSTM, CNN, or Transformer.

🔷

2. Context Vector (c)

A function of all encoder hidden states h1:n. Conveys the essence of the entire input to the decoder. This is the "bottleneck."

📤

3. Decoder

Accepts context vector c, generates output sequence y1:m of arbitrary length autoregressively, one token at a time.

How the Encoder Works

In practice, encoders are stacked biLSTMs (bidirectional LSTMs). The top-layer forward and backward hidden states are concatenated to produce the contextualized representation for each input time step. The final hidden state hn becomes the context vector C.

How the Decoder Works

The decoder uses C to initialize its hidden state and then generates tokens autoregressively. To prevent the context from "fading," C is also concatenated at every decoding step:

DECODER HIDDEN STATE UPDATE hi = g(hi-1, ŷi-1, C)

Where: g = RNN function, ŷi-1 = embedding of previous output token, C = context vector

Training: End-to-End with Teacher Forcing

Teacher Forcing

During training, instead of feeding the decoder's own (possibly wrong) output back as the next input, we feed the gold (correct) target token. This prevents error accumulation and speeds up training. During inference, the model uses its own predictions.

During inference the decoder uses its predicted output ŷ as the next input. During training with teacher forcing it uses the true label. This mismatch is a known limitation called "exposure bias."

Language Model Formulation

An encoder-decoder translation model simply concatenates the source text, a separator token <s>, and the target text, then computes:

CONDITIONAL PROBABILITY p(Y|X) = ∏ p(yi | y<i, X)

Limitations of Basic Encoder-Decoder

  • Fixed-length bottleneck: The entire source must be compressed into one fixed-size vector C — impossible for long sequences.
  • No selective focus: No mechanism to attend to specific relevant parts of the input while generating each output token.

Attention Mechanism

Attention Mechanism

A solution to the bottleneck problem. Instead of relying on a single static context vector, the decoder computes a dynamic context vector Ci at each decoding step i, as a weighted sum of ALL encoder hidden states. Each weight reflects how relevant each encoder state is to the current decoding step.

Step-by-Step: How Attention is Computed

ALGORITHM

Dot-Product Attention — 4 Steps

Step 1 — Score: Compute relevance of each encoder state hj to the current decoder state h̃i:
score(h̃i, hj) = h̃i · hj (dot product)
Step 2 — Normalize (Softmax): Convert scores to a probability distribution:
αij = softmax(score(h̃i, hj)) ∀j
αij is the attention weight: how much attention decoder step i pays to encoder step j.
Step 3 — Weighted Sum (Context Vector):
Ci = Σj αij · hj
This gives a dynamic, fixed-length summary of the encoder states, focused on what's relevant now.
Step 4 — Decode: Use Ci to compute the next hidden state and generate a token:
hi = g(hi-1, ŷi-1, Ci)

Types of Attention

Type How Score is Computed Speed Flexibility
Bahdanau (Additive) Uses a small neural network layer: score = W·tanh(W1·h̃ + W2·h) Slower Very flexible
Luong (Multiplicative) Simple dot product: score = h̃ · h Faster Memory-efficient
Global Attention Considers ALL encoder hidden states Slower for long seqs Most thorough
Local Attention Focuses on a small window around current position Faster Good for long docs

Why RNN-based Attention Still Has Limits

Even with attention, RNNs are inherently sequential: step 100 cannot be computed until steps 1–99 are done. This prevents parallelization on GPUs, making training slow. → Solution: Transformers.

Transformers & Self-Attention

Transformer

A neural architecture that uses self-attention instead of recurrence. It is highly parallelizable and scales efficiently. Transformers map input sequences (x1,…,xn) to output sequences (z1,…,zn) of the same length using stacked transformer blocks.

Self-Attention

An attention mechanism where a sequence attends to itself. Each token in a sequence looks at all other tokens (in its context window) to build a richer representation. Unlike encoder-decoder attention, self-attention operates within a single sequence.

Query, Key, Value — The Core Idea

In self-attention, each token plays three roles simultaneously:

🔍

Query (Q)

The current token being processed. "What am I looking for?"

🗝️

Key (K)

Every token in the context, used for matching. "What do I contain?"

💎

Value (V)

The actual content contribution of each token to the output.

Scaled Dot-Product Self-Attention

SELF-ATTENTION FORMULA (EQUATIONS 10.4 → 10.12) Step 1 — Scores: score(qi, kj) = qi · kj

Step 2 — Scale: score_scaled = (qi · kj) / √dk

Step 3 — Softmax: αij = softmax(score_scaled)

Step 4 — Output: ai = Σj αij · vj

In matrix form: Attention(Q,K,V) = softmax(QKT / √dk) V
We divide by √dk to prevent extremely large dot products from pushing softmax into regions of near-zero gradients (vanishing gradient during training).

Causal (Backward-Looking) Self-Attention

Used in language models (like GPT). At position i, the token can only attend to positions ≤ i (no future context). This is implemented by masking future positions with -∞ before softmax. Enables autoregressive generation.

Bidirectional Self-Attention

Used in encoders (like BERT). Each token can attend to all other tokens — both past and future. Better for tasks that need full context understanding (classification, NER, fill-in-the-blank).

Multi-Head Attention

Multi-Head Attention

Instead of one self-attention computation, we run h parallel attention heads, each with its own Q, K, V weight matrices. Each head can learn different types of relationships. The outputs are concatenated and projected back.

MULTI-HEAD ATTENTION MultiHead(Q, K, V) = Concat(head1, ..., headh) WO
where headi = Attention(QWiQ, KWiK, VWiV)

The Transformer Block

Each transformer block contains 4 components in order:

1. Multi-Head Self-Attention Layer
↓
2. Add & Norm (Residual Connection + Layer Norm)
↓
3. Feed-Forward Network (two linear layers + ReLU)
↓
4. Add & Norm (Residual Connection + Layer Norm)

Positional Encoding

Since self-attention has no inherent notion of order (unlike RNNs), we add positional encodings to the input embeddings to inject sequence position information. Without this, the model treats all positions identically.

Transformer Models by Architecture Type

Type Examples Attention Type Good For Cannot Do
Encoder-Only BERT, RoBERTa, DistilBERT Bidirectional Classification, NER, fill-in-blank Generate text, translate, summarize
Decoder-Only GPT, GPT-2, GPT-3, LLaMA Causal (unidirectional) Text generation, chatbots, code Bidirectional understanding tasks
Encoder-Decoder BART, T5 Both Translation, summarization, Q&A —

Transfer Learning & Pre-training

Pre-training

Training a model on large amounts of unlabeled data with self-supervised objectives (e.g., next token prediction, masked language modeling). Weights start random and are learned from massive corpora. Produces a "base model" with general language understanding.

Transfer Learning

Taking a pre-trained model (already trained on large data) and adapting it to a new, specific task using a much smaller labeled dataset. Avoids training from scratch. The model "transfers" its general knowledge to the new domain.

Benefits of Transfer Learning

⚡

Faster Development

No need to train from scratch. Start from a powerful baseline and fine-tune quickly.

📉

Less Data Needed

Fine-tuning only requires a small labeled dataset — the base model supplies general knowledge.

🏆

Better Results

Pre-trained models encode rich representations from billions of examples, leading to superior task performance.

Types of Deep Transfer Learning

Type Description Example
Instance-based Reuse specific training instances from source domain Weighting source samples
Representation-based Learn shared feature representations between domains Shared embeddings
Model Sharing-based Share parameters or model layers between tasks VGG-16 on ImageNet → custom task
Adversarial-based Use adversarial training to align source/target domains Domain adaptation GANs

Implementation: Model Sharing (CNN Example)

Using VGG-16 (pre-trained on ImageNet with 1M+ images) and fine-tuning for a custom image classification task. The early layers (edge detectors, texture filters) are frozen; the final layers are retrained.

GloVe: Pre-trained Word Embeddings

GloVe (Global Vectors for Word Representation)

An unsupervised algorithm that learns word vector representations by training on global word-word co-occurrence statistics from a corpus. The resulting vectors capture semantic relationships (e.g., king − man + woman ≈ queen). GloVe vectors represent a form of transfer learning for NLP.

Beam Search

Beam Search

A heuristic search algorithm that maintains k (beam width) active candidate sequences at each decoding step. At every step, it expands each candidate by all possible next tokens, scores the resulting W×B candidates, and keeps only the top k. It is a middle ground between greedy search (k=1) and exhaustive search (k=∞).

Why Greedy Decoding Fails

Greedy decoding picks the highest-probability token at each step (argmax). Problem: a high-probability first token may lead to a very low-probability overall sequence. It cannot backtrack — it's "short-sighted" and gets trapped in local optima. It's also deterministic and produces repetitive text.

GREEDY DECODING ŷt = argmax P(yt | y1, ..., yt-1, X)

The Search Space Problem

Greedy (k=1)
Fast, local optima
→
Beam Search (k=5–10)
Best balance
→
Exhaustive (k=∞)
Infeasible

Beam Search Algorithm — Step by Step

ALGORITHM

Beam Search with W=2, B=3

1. Initialization: Start with root node (start token). Generate B=3 successors.
2. Expansion: From the W active beams, generate all B successors. Total candidates = W × B = 6.
3. Selection: Score all 6 candidates using the heuristic (log probability sum). Keep top W=2.
4. Iteration: Repeat Steps 2–3 until end-of-sequence token generated or max length reached.
5. Termination: Return the highest-scoring complete sequence among all beams.

Key Characteristics

Property Details
Beam Width (W) Number of candidates kept at each level. Higher W = better quality but slower.
Branching Factor (B) Number of possible next tokens (vocab size V). Evaluates W×B at each step.
Time Complexity O(T × V × k) where T = sequence length, V = vocabulary size, k = beam width
k=1 Equivalent to Greedy Search
k→∞ Approaches Exhaustive Search

ADVANTAGES

  • More efficient than exhaustive search
  • Better quality than greedy
  • Adjustable via beam width hyperparameter
  • Memory bounded by W

LIMITATIONS

  • No guarantee of optimal solution
  • Biased toward "safe," high-probability phrases
  • Produces repetitive, generic text
  • Heuristic-dependent quality

When to Use Beam Search vs. Sampling

Task Type Preferred Method Reason
Machine Translation Beam Search Deterministic, constrained — one correct answer
Summarization Beam Search Factual accuracy important
Chatbots / Storytelling Sampling (top-p, temperature) Diversity and creativity needed
Code Generation Beam Search or Greedy Correctness over diversity

Retrieval-Augmented Generation (RAG)

RAG (Retrieval-Augmented Generation)

A technique that enhances LLMs by combining them with a retrieval system that fetches relevant external information at inference time. Instead of relying solely on internal (parametric) knowledge, the model also uses dynamically retrieved (non-parametric) external knowledge. Reduces hallucinations and keeps information current without retraining.

Why RAG? The Problem with Pure LLMs

  • Static knowledge: LLMs are frozen at training cutoff. They can't answer questions about recent events.
  • Hallucinations: Without retrieval, LLMs make up plausible-sounding but wrong facts.
  • No provenance: Can't cite sources or explain where information came from.
  • Fine-tuning is expensive: Updating a model for new domain knowledge requires costly retraining.

Parametric vs. Non-Parametric Memory

🧠

Parametric Memory

Knowledge stored inside the model's weights (parameters) learned during training. Fixed after training ends.

🗄️

Non-Parametric Memory

External knowledge base (vector database of documents). Can be updated without retraining the model.

RAG Pipeline — How It Works

PIPELINE

RAG Process Flow

1. Query: User submits a question or prompt.
2. NLP Analysis: System parses query to understand intent.
3. Embedding: Query is converted to a vector embedding.
4. Similarity Search: Vector database is searched for documents with similar embeddings.
5. Ranking & Filtering: Retrieved documents ranked by relevance; top K selected.
6. Augmented Prompt: Original query + retrieved context → combined into enriched prompt.
7. Generation: LLM generates answer using both its parametric knowledge and retrieved context.
8. Post-processing (optional): Fact-checking, summarization, formatting of the response.

Two RAG Model Variants

📄

RAG-Sequence

Uses the same retrieved document to predict every target token in the output. One document for the whole answer.

🔀

RAG-Token

Can use a different retrieved document for each target token. More flexible, can pull from multiple sources per answer.

Chunking

Chunking

Breaking large documents into smaller, manageable pieces before storing in the vector database. Improves retrieval accuracy because smaller units can be matched more precisely to specific queries.

ADVANTAGES OF RAG

  • Reduces hallucinations with grounded facts
  • Provides citations and provenance
  • Access to current/real-time information
  • No expensive model retraining for new data
  • Transparent reasoning process
  • Handles ambiguous queries with context

LIMITATIONS OF RAG

  • Missing info if KB is incomplete
  • Suboptimal retrieval may miss relevant docs
  • Token limit constraints on retrieved context
  • LLM may still hallucinate despite context
  • Complex system infrastructure to maintain
  • Bias propagation from knowledge base

LLM Full Training Pipeline

Modern LLMs are not simply trained once. They go through a 3-stage progressive pipeline, each building on the previous.

Stage 1: Pre-training
Trillions of tokens • Self-supervised • Next token prediction
Stage 2: Supervised Fine-Tuning (SFT)
Instruction-following pairs • Smaller curated dataset
Stage 3: Preference Tuning (RLHF / DPO)
Human feedback • Helpful, Honest, Harmless

Stage 1: Pre-training

Pre-training

The most computationally expensive phase. Uses Self-Supervised Learning (SSL): the model predicts the next token from a massive unlabeled text corpus (Wikipedia, books, GitHub, ArXiv, Common Crawl). No human labels needed — the text itself is the supervision.

The model learns: grammar, world facts, reasoning, coding conventions, and much more. But a pre-trained model is NOT an assistant — if asked "What is the capital of France?" it might respond "And what is the capital of Germany?" because it thinks it's completing a quiz.

Stage 2: Supervised Fine-Tuning (SFT)

SFT / Instruction Tuning

Training on a smaller, high-quality dataset of (Prompt, Response) pairs written by humans. Teaches the model the format of a helpful conversation. After SFT, the model knows that a question should be answered directly.

Data mixtures include: assistant dialogs, synthetic instructions, math/reasoning/code problems, safety alignment examples. Data can be both human-written and synthetic.

Stage 3: Preference Tuning

RLHF (Reinforcement Learning from Human Feedback)

ALGORITHM

RLHF Process

1. Sampling: The fine-tuned model generates multiple responses to the same prompt.
2. Ranking: Human raters rank responses from best to worst (helpfulness, accuracy, harmlessness).
3. Reward Model: A separate smaller model is trained to predict human preferences based on the rankings.
4. Optimization: The main LLM is updated using PPO (Proximal Policy Optimization) to maximize the reward model's score.

DPO (Direct Preference Optimization)

A newer, more efficient alternative to RLHF. Skips the separate reward model and directly optimizes the LLM on ranked pairs (chosen vs. rejected responses). More stable training, lower computational cost.

Scaling Laws

Scaling Laws

Performance of LLMs follows predictable power-law relationships with three factors: Parameters (N), Data (D), and Compute (C). Discovered by OpenAI and DeepMind researchers.

Chinchilla Observation: For optimal training, dataset size should grow proportionally with model size. Scaling parameters without scaling data leads to diminishing returns quickly.

LoRA: Parameter-Efficient Fine-Tuning

LoRA (Low-Rank Adaptation)

Instead of fine-tuning all model weights (expensive), LoRA adds small, trainable low-rank matrices to each layer. Only a small fraction of parameters are updated, with similar performance to full fine-tuning. Key benefit: you can swap LoRA matrices to switch tasks without re-loading the base model.

LoRA needs a higher learning rate than full fine-tuning, and performs poorly with large batch sizes compared to full fine-tuning.

Dialogue Systems & Conversational Agents

Conversational Agents / Dialogue Systems

Systems designed to interact with humans via text or speech. Examples: SIRI, Alexa, Cortana, Google Assistant. They range from simple (timers, music) to complex (travel booking, mental health support).

Two Types of Conversational Agents

💬

Chatbots

Mimic informal human conversation. Goal: engaging, natural dialogue for fun or therapy. Example: early ELIZA, modern LLM chatbots.

🎯

Task-Based Dialogue Agents

Goal: help user complete a specific task (booking flights, setting alarms, ordering food). Architecture: frames with slots and values.

Chatbot Architectures

Type Examples Mechanism
Rule-Based: Pattern-Action ELIZA Pattern matching → scripted responses
Rule-Based: Mental Model PARRY Internal state machine + rules. First to pass Turing Test!
Corpus: Information Retrieval XiaoIce Retrieve best matching response from past conversations
Corpus: Neural Encoder-Decoder BlenderBot Generate response from user utterance using seq2seq model

Task-Based Dialogue: Frames and Slots

Frame

A knowledge structure representing user intent, consisting of slots (fields to be filled) and associated questions. Example for flight booking:

FLIGHT BOOKING FRAME SLOT TYPE QUESTION ORIGIN city "What city are you leaving from?" DEST city "Where are you going?" DEP_DATE date "What day would you like to leave?" DEP_TIME time "What time would you like to leave?" AIRLINE line "What is your preferred airline?"

Properties of Human Conversation

  • Turns: Each contribution to a conversation is a "turn." Conversation is a game where participants take turns.
  • Turn-taking: Rules for when to take the floor and when to yield it. Includes interruptions.
  • Barge-in: Allowing a user to interrupt the system mid-speech.
  • End-pointing: Detecting when the user has stopped talking. Very hard — people pause in the middle of turns.
  • Latching: Speakers time their response to begin immediately when the other finishes, with minimal pause.

Speech Acts (Dialogue Acts)

Speech Acts

Each utterance in dialogue is an action. Classification from Bach and Harnish (1979):

Constatives

Commit speaker to something being true. (answering, claiming, stating, denying)

Directives

Speaker tries to get addressee to do something. (asking, ordering, requesting, advising)

Commissives

Commit speaker to future action. (promising, planning, vowing, betting)

Acknowledgments

Express speaker's attitude toward hearer. (greeting, thanking, apologizing)

Grounding

Grounding

The process of establishing common ground — mutual confirmation that communication was understood. The Principle of Closure: agents need evidence, sufficient for current purposes, that they succeeded in performing an action. In dialogue: one speaker signals understanding (e.g., "OK", "uh-huh"), the other continues.

Example: Why do elevator buttons light up? To ground the action — confirm to the user that pressing was registered.

Conversation Structure

  • Adjacency Pairs: Locally structured pairs: QUESTION→ANSWER, PROPOSAL→ACCEPTANCE/REJECTION, COMPLIMENT→DOWNPLAYER
  • Subdialogues: Embedded sub-conversations within the main dialogue (e.g., clarification subdialogue, correction subdialogue)
  • Presequences: "Can you make train reservations?" → "Yes" → "Then I'd like to book..."

Mixture of Experts (MoE)

Mixture of Experts (MoE)

An architecture that improves LLM efficiency by dividing model capacity among multiple specialized sub-networks called "experts." A learned router selects only a small subset of experts for each input token. Unlike dense models (all parameters active every time), MoE models are sparse — most parameters sleep while only relevant ones activate.

Why MoE?

Dense models (e.g., LLaMA 3) activate all parameters for every token — even parameters specialized for chemistry are activated when processing a cooking recipe. MoE makes this efficient by routing tokens only to relevant experts.

💡

Scale without Cost

MoE models can have 100B–1T+ parameters while keeping per-token compute manageable, because only ~1–2 experts activate per token.

🔀

Specialization

Different experts learn different aspects of language/knowledge. One expert may specialize in code, another in mathematics.

Routing Collapse — Key Challenge

The most dangerous MoE training failure: the router always sends tokens to the same 1–2 experts, ignoring all others. The unused experts never learn. Called routing collapse. Solved with auxiliary load-balancing losses that penalize imbalanced routing.

MoE in Transformers

In transformer-based MoE (e.g., Mixtral), the feedforward network (FFN) layer within each transformer block is replaced by a mixture of FFN experts plus a routing network. The attention layers remain shared/dense.

Decoding & Sampling Strategies

After the LLM computes probability distributions over the vocabulary, a decoding strategy determines which token to pick next.

Strategy How It Works Problem Best For
Greedy Always pick argmax P(yt|context) Repetitive, deterministic, local optima Simple tasks
Beam Search Keep top-k sequences at each step Generic, "safe" output; no true creativity Translation, summarization
Random Sampling Sample from full distribution P(yt|context) Low-probability words pollute output Not recommended directly
Top-k Sampling Sample only from top-k highest probability tokens k is fixed regardless of distribution shape Creative tasks
Top-p (Nucleus) Sampling Sample from smallest set covering cumulative probability ≥ p More complex to tune Best for open-ended generation
Temperature Sampling Reshape distribution: divide logits by T before softmax T tuning required Controlling diversity

Temperature Sampling Deep Dive

TEMPERATURE SCALING P'(yt) = softmax(logits / T)

T < 1.0 → sharper distribution → model is more confident → less diversity T = 1.0 → original distribution T > 1.0 → flatter distribution → model is more random → more diversity

Top-p (Nucleus) Sampling vs. Top-k

Problem with top-k: If k=10, sometimes the top 10 tokens cover 99% of probability mass (good), other times they cover only 30% (the rest is spread thinly — bad). k doesn't adapt to distribution shape.

Top-p fix: Include the smallest set of tokens whose cumulative probability ≥ p (e.g., 0.9). Automatically adapts the number of candidates to the distribution.

Inference Optimizations

  • KV Caching: Cache Key and Value tensors from attention for previously processed tokens. Avoids recomputing them at every decoding step.
  • FlashAttention: Optimizes memory access patterns in GPU — reduces reads/writes between high-bandwidth memory (HBM) and on-chip SRAM.
  • PagedAttention: Stores K and V in non-contiguous memory pages (like OS virtual memory). Minimizes wasted memory from fragmentation.
  • Multi-Query Attention / Grouped Query Attention: Share key/value heads across groups of query heads — reduces KV cache size.

PROBLEM 1

Trace Beam Search Step-by-Step

Scenario: You are decoding with beam width k=2. The vocabulary is {A, B, C}. Log-probabilities at each step are given below.

STEP 1: Initial probabilities P(A | <start>) = 0.5 → log = -0.69 P(B | <start>) = 0.3 → log = -1.20 P(C | <start>) = 0.2 → log = -1.61
STEP 2: Given beam survived ["A", "B"], conditional log-probs P(A|A)=0.1, P(B|A)=0.6, P(C|A)=0.3 P(A|B)=0.5, P(B|B)=0.2, P(C|B)=0.3

Task:

  1. After Step 1: Which 2 tokens survive? (Keep k=2 highest log-prob candidates)
  2. After Step 2: Compute all 6 candidate sequences (2 beams × 3 tokens). Sum log-probs. Which 2 sequences survive?
  3. What is the difference between this and greedy decoding?
WORKED SOLUTION — STEP 1 Survivors: A (log=-0.69), B (log=-1.20) ✓ [C dropped] STEP 2: Compute joint log-probs A→A: -0.69 + log(0.1) = -0.69 + (-2.30) = -2.99 A→B: -0.69 + log(0.6) = -0.69 + (-0.51) = -1.20 ★ A→C: -0.69 + log(0.3) = -0.69 + (-1.20) = -1.89 B→A: -1.20 + log(0.5) = -1.20 + (-0.69) = -1.89 B→B: -1.20 + log(0.2) = -1.20 + (-1.61) = -2.81 B→C: -1.20 + log(0.3) = -1.20 + (-1.20) = -2.40 Survivors: "AB" (-1.20) and "AC" (-1.89) ★ Greedy would have chosen: A (step 1) → B (step 2) = "AB" Beam search found same here, but on longer sequences it diverges!
PROBLEM 2

Compute Dot-Product Attention

Scenario: You have an encoder with 3 hidden states and a decoder state at step 1:

VECTORS (simplified 2D for practice) Encoder: h1 = [1, 0] h2 = [0, 1] h3 = [1, 1] Decoder: d1 = [1, 0.5]

Task:

  1. Compute the score (dot product) of d1 with each encoder hidden state.
  2. Apply softmax to get attention weights α.
  3. Compute context vector C = Σ αj · hj.
  4. Interpret: which encoder position did the decoder attend to most?
WORKED SOLUTION Step 1 — Scores (dot products): score(d1, h1) = 1·1 + 0.5·0 = 1.0 score(d1, h2) = 1·0 + 0.5·1 = 0.5 score(d1, h3) = 1·1 + 0.5·1 = 1.5 ← highest Step 2 — Softmax (exp then normalize): exp(1.0) = 2.718, exp(0.5) = 1.649, exp(1.5) = 4.482 sum = 8.849 α1 = 2.718/8.849 ≈ 0.307 α2 = 1.649/8.849 ≈ 0.186 α3 = 4.482/8.849 ≈ 0.507 Step 3 — Context vector: C = 0.307·[1,0] + 0.186·[0,1] + 0.507·[1,1] = [0.307,0] + [0,0.186] + [0.507,0.507] = [0.814, 0.693] Step 4 — Interpretation: The decoder attends MOST to h3 (α=0.507), moderately to h1, least to h2.
PROBLEM 3

Temperature Sampling Calculation

Scenario: A language model produces the following raw logits for the next token:

Logits: token_A = 3.0, token_B = 1.0, token_C = 0.5

Task:

  1. Compute the softmax probabilities at T=1.0 (standard).
  2. Compute the softmax probabilities at T=0.5 (cold/sharp).

RAG vs. Standard LLM: Design Problem

Scenario: You are building a Q&A system for a hospital. Users ask questions about hospital-specific protocols and recent drug approvals (updated monthly).

  1. Would you use a standard LLM or RAG? Justify with at least 3 specific reasons.

MODEL ANSWER 1. Use RAG because: - Hospital protocols change monthly → LLM training data is stale - Medical hallucinations are dangerous → need grounded, citable facts - Hospital-specific protocols can't be in general training data - Updating KB is cheaper than retraining a large model 2. Knowledge base contents: - Hospital-specific protocol PDFs - Drug approval databases (FDA, EMA), updated monthly - Patient care guidelines, formulary lists - Internal clinical research summaries 3. Failure scenarios: - Query-document mismatch: "What's the max dose of X?" ≠ "Maximum dosage for X" → use better embedding/chunking strategies - Missing information: drug not yet in KB → system must say "unknown" not hallucinate a dose - Token limit: long protocols may be truncated → smart chunking needed 4. RAG-Token preferred here: - Medical answers often synthesize from multiple sources (protocol doc + drug database + research paper) - Each fact in the answer may come from a different source - RAG-Sequence would be limited to a single retrieved document
